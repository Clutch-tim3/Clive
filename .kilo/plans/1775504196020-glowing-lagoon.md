# Name.com Domain Service Implementation Plan

## Overview
Replace the current broken ResellerClub/RDAP domain integration with Name.com v4 API for real domain availability checking and registration.

## Current State Analysis

### Already Implemented (✓)
1. **lib/domains/namecom.ts** - Complete Name.com API wrapper exists and matches the prompt
2. **lib/domains/availability.ts** - Contains `parseDomainInput` and `checkAllTLDs` functions
3. **lib/domains/tlds.ts** - Contains `FEATURED_TLDS`, price caching, and USD→ZAR conversion
4. **app/api/domains/check/route.ts** - Uses correct availability functions from prompt
5. **app/api/domains/tlds/route.ts** - Returns TLD prices

### Needs to Be Fixed (✗)
1. **app/api/domains/register/route.ts** - Still uses ResellerClub instead of Name.com
2. **app/api/domains/[domain]/route.ts** - Needs to work with Name.com
3. **app/api/domains/[domain]/renew/route.ts** - Needs to work with Name.com
4. **app/api/domains/renew/route.ts** - Duplicate route, needs consolidation
5. **.env.local** - Contains working Name.com credentials that need verification

## Implementation Steps

### Step 1: Verify Credentials and Environment
- Check current Name.com credentials in `.env.local`
- Verify `NAMECOM_USERNAME` and `NAMECOM_API_TOKEN` are set correctly
- Ensure `NAMECOM_ENV=production` is set

### Step 2: Update Register API (app/api/domains/register/route.ts)
Replace the ResellerClub implementation with Name.com:
- Import `searchDomains`, `registerDomain`, `lockDomain`, `buildContact`, `NamecomError` from `lib/domains/namecom`
- Import `getTLDPrice`, `usdToZAR` from `lib/domains/tlds`
- Add live availability check before registration
- Build contact using `buildContact` helper
- Call Name.com API for registration
- Lock domain after registration
- Write to Firestore with correct schema (ncOrderId, priceUSD, priceZAR, etc.)

### Step 3: Update Domain Detail API (app/api/domains/[domain]/route.ts)
- Implement GET for domain details
- Implement PUT for updating settings (nameservers, autoRenew, lock)

### Step 4: Update Domain Renew API (app/api/domains/[domain]/renew/route.ts)
- Implement POST for domain renewal
- Get current price from Name.com API
- Update Firestore with new expiry

### Step 5: Remove Duplicate Routes
- Remove or consolidate `app/api/domains/renew/route.ts` if redundant

### Step 6: Update Console Integration
- Ensure domains tab works with new Firestore schema
- Update badge count logic if needed

### Step 7: Test the Integration
- Test domain search to verify Name.com API calls work
- Test registration flow (will fail without real credentials or if domain is taken)
- Verify error handling

## Firestore Schema

The domain document should include:
- `domainName`, `tld`, `years`, `status`
- `locked` (transfer lock - ON by default)
- `autoRenew`, `privacyEnabled`, `nameservers`
- `priceUSD`, `priceZAR` (at time of purchase)
- `ncOrderId` (Name.com order reference)
- `contact` object with registrant details
- `registeredAt`, `expiresAt`, `renewedAt`, `createdAt`, `updatedAt`

## Error Handling

| Scenario | User Sees |
|---|---|
| API unavailable | "Domain service temporarily unavailable" |
| Domain taken | Red error: "This domain is no longer available" |
| Auth failed | "Domain service temporarily unavailable" (log server-side) |
| Registration success | Redirect to /domains/dashboard |
